# GPU Hardware Intrinsic Profiling Agent

This repository now contains both course phases:

- Phase 1: GPU hardware intrinsic probing and kernel bottleneck analysis
- Phase 2: An agentic LoRA kernel optimizer that maintains `optimized_lora.cu`

An autonomous, LLM-powered agent for probing GPU hardware characteristics via CUDA micro-benchmarks, cross-verifying with NVIDIA Nsight Compute (ncu), and performing kernel-level bottleneck analysis — all driven by a single `target_spec.json`.

All CUDA micro-benchmark probe source code is **generated autonomously by the LLM** at runtime; no pre-written benchmark files are required or used.

## Phase 2: LoRA Operator Optimization

### Problem Statement

Phase 2 optimizes the LoRA forward operator

$$
Y = W @ X + A @ (B^T @ X)
$$

under the public workload:

- hidden size `d ∈ [3584, 4608]`
- LoRA rank `r = 16`
- strict correctness gate `torch.allclose(rtol=1e-4, atol=1e-4)`
- target platform `NVIDIA GeForce RTX 3090`

The submission artifact is `optimized_lora.cu`, regenerated by the Phase 2 agent and executed via `bash run.sh`.

### Agent Design

The final optimizer is intentionally template-based rather than LLM-authored at kernel generation time. After earlier free-form codegen experiments proved brittle, the search space was frozen into eight deterministic variants so every run compiles the same candidate set, validates correctness, benchmarks on the three public sizes, and writes the best correct implementation back to `optimized_lora.cu`.

```text
bash run.sh
  |
  v
agent.py --phase phase2
  |
  v
LoRAOptimizationAgent
  |
  +--> fixed candidate library (8 variants)
  |      |
  |      +--> render CUDA template
  |      +--> compile PyTorch CUDA extension
  |      +--> validate vs PyTorch reference
  |      +--> benchmark d=3584,4096,4608
  |      +--> record score / failure reason
  |
  +--> keep best correct candidate
  |
  +--> write optimized_lora.cu
  +--> write phase2_results.json
  +--> write phase2_summary.md
```

Why this approach was kept for the final submission:

- deterministic candidate set for reproducible engineering evidence
- direct compile/benchmark loop with no hidden codegen variability
- easy to explain why each candidate exists and why it won or lost
- robust under strict FP32 correctness constraints

### Final 8-Variant Results

These are the validated results from the final post-cleanup full search.

| Variant | Strategy | Result |
|---------|----------|--------|
| `cublas_3gemm_fp32` | Exact FP32 cuBLAS baseline | `1.0218x` |
| `cublas_streams_overlap` | Overlap `W@X` and `B^T@X` on two streams, then `A@tmp` | `1.0287x` |
| `fused_lora_update_reg16_bs16` | Register-resident rank-16 fused update | `0.9969x` |
| `fused_lora_update_reg16_bs32` | Larger register-tiled fused update | `0.9792x` |
| `fused_lora_float4_bs16` | Float4-vectorized fused update | `1.0224x` |
| `fused_lora_float4_bs32` | Larger float4-vectorized fused update | `1.0156x` |
| `wmma_fp16_accum_f32` | WMMA/Tensor Core `W@X`, FP32 low-rank path | `FAIL` |
| `wmma_fp16_streams` | WMMA/Tensor Core `W@X` plus stream overlap | `FAIL` |

### Why Each Variant Was Tried

| Variant | Why it was tried | What was learned |
|---------|------------------|------------------|
| `cublas_3gemm_fp32` | Establish the exact FP32 cuBLAS baseline. | cuBLAS already sits close to the hardware limit; any gain must come from orchestration, not replacing the dense GEMM. |
| `cublas_streams_overlap` | Overlap the dominant dense GEMM with the small low-rank temporary GEMM. | This was the best valid design because it preserved exact FP32 arithmetic while hiding part of the low-rank work. |
| `fused_lora_update_reg16_bs16` | Remove the final `A@tmp` GEMM with a tiny custom kernel. | The low-rank epilogue is too small to matter much; custom SIMT code lost against cuBLAS quality. |
| `fused_lora_update_reg16_bs32` | Test whether a larger tile improves fused update throughput. | Larger tiles increased pressure without overcoming the dense GEMM bottleneck. |
| `fused_lora_float4_bs16` | Increase memory efficiency of the fused update using vectorized loads and stores. | This was the best custom-kernel attempt, but still slightly behind the overlapped cuBLAS design. |
| `fused_lora_float4_bs32` | Check whether larger vector tiles help the float4 variant. | The larger tile again regressed slightly relative to the smaller vectorized kernel. |
| `wmma_fp16_accum_f32` | Use Tensor Cores for `W@X` while keeping the low-rank path exact. | FP16 input quantization hit a hard correctness wall despite FP32 accumulation. |
| `wmma_fp16_streams` | Combine Tensor Cores with overlap, chasing both compute and scheduling gains. | The same FP16 precision wall disqualified the approach before speed mattered. |

### Key Findings

- Stream overlap gave the best valid result. The final end-to-end run measured `1.0287x`, while repeated reruns stayed in the same `~1.027x` band.
- The dense `W@X` GEMM dominates runtime. Low-rank fusion changes only the margins because the rank is fixed at `16`.
- `wmma_fp16_accum_f32` and `wmma_fp16_streams` both failed the `atol=1e-4` gate with `max_abs_err ≈ 0.10647583`. The limiting factor was FP16 input truncation, not the WMMA layout.
- A follow-up experiment replaced the explicit `B^T` materialization with in-place cuBLAS transpose using `CUBLAS_OP_T`. That path was mathematically correct but still failed the same correctness gate with `max_abs_err ≈ 7.8e-4`, which is consistent with a different FP32 accumulation order landing outside the strict tolerance boundary.
- Removing redundant template-side `.contiguous()` calls did not change the winning strategy. The templates now require contiguous inputs explicitly via `TORCH_CHECK` and operate directly on the supplied tensors.

### Conclusion

On this RTX 3090 environment, `cublas_streams_overlap` is the best fully valid FP32 implementation found by the fixed 8-variant search, with repeated runs landing around `1.027x` to `1.029x` over the PyTorch reference. The remaining Tensor Core and transpose-in-place experiments both crossed the strict `atol=1e-4` correctness boundary, so this narrow band is the practical FP32 SIMT ceiling for this workload on this hardware.

### How To Run

```bash
bash run.sh
```

The Phase 2 agent writes:

- `optimized_lora.cu` — the current best single-file implementation
- `phase2_results.json` — candidate-by-candidate benchmark history
- `phase2_summary.md` — compact optimization summary

### target_spec.json Format

```jsonc
{
  "targets": ["dram_latency_cycles", "actual_boost_clock_mhz", ...],
  "run": "build/kernels/matmul_naive"   // optional: kernel binary to profile
}
```

| Field | Purpose | Required |
|-------|---------|----------|
| `targets` | Hardware metrics to measure via micro-benchmarks (Section 1.7) | No |
| `run` | CUDA executable to profile for bottleneck analysis (Sections 1.1–1.6) | No |

Both fields are optional — the agent runs whichever phases are requested. Target names are **not hardcoded**: unknown metric names are resolved at runtime via an LLM-based semantic classifier.

## Phase 1: Hardware Intrinsic Probing (Section 1.7)

### Supported Metrics

| Metric | Probe | Measurement Method |
|--------|-------|-------------------|
| `l1_latency_cycles` | latency_probe | Pointer chasing, array ≤ 16 KB (L1-resident) |
| `l2_latency_cycles` | latency_probe | Pointer chasing, L1 < array < L2 boundary |
| `dram_latency_cycles` | latency_probe | Pointer chasing, array ≫ L2 capacity |
| `l2_cache_size_kb` | latency_probe | Latency curve inflection (cliff) detection |
| `max_global_bandwidth_gbps` | bandwidth_probe | float4 coalesced reads, all SMs, max of sweep |
| `max_shmem_bandwidth_gbps` | bandwidth_probe | Bank-conflict-free shmem reads, per-SM + aggregate |
| `actual_boost_clock_mhz` | clock_probe | clock64() / CUDA-events under sustained FMA load |
| `num_active_sms` | clock_probe | Inline PTX `%smid` register, unique-ID counting |
| `max_shmem_per_block_kb` | shmem_limit_probe | Binary search + `cudaFuncSetAttribute` opt-in |
| `bank_conflict_penalty_cycles` | bank_conflict_probe | Stride-1 (0 conflicts) vs stride-32 (32-way) |

> **Autonomous probe generation**: all CUDA source code for the probes above is written by the LLM at runtime based on design specifications. Sources are cached in `build/*_generated.cu` and reused on subsequent runs to avoid redundant API calls.

### LLM-Based Semantic Target Resolution

Unknown metric names (e.g., `"mem_delay"` instead of `"dram_latency_cycles"`) are batch-resolved in a single LLM call to the correct probe. Results are cached per session.

### Multi-Strategy Cross-Verification

Every measurement is verified using at least two independent methods:

| Strategy | Role |
|----------|------|
| **CUDA micro-benchmark** | Primary measurement (hardware-level, immune to API spoofing) |
| **ncu profiling** | Secondary verification (DRAM throughput → implied bandwidth; cycle count → clock) |
| **nvidia-smi / CUDA API** | Tertiary sanity check (device properties, clocks, SM count) |

### Anti-Hacking Resilience

| Threat | Defence |
|--------|---------|
| **Frequency Locking** | Clock measured via `clock64()`/CUDA-events ratio — no API dependency |
| **SM Masking** | Active SMs detected via inline PTX `%smid` — bypasses virtualisation |
| **Spoofed Device Properties** | All primary data from micro-benchmarks; API values used only for comparison |
| **Prefetcher Interference** | Pointer-chasing with random stride defeats hardware prefetching |

## Phase 1: Kernel Bottleneck Analysis (Sections 1.1–1.6)

When `target_spec.json` contains a `"run"` field (or `--kernel` is passed), the agent performs the full 4-step analysis:

| Step | Section | What It Does |
|------|---------|-------------|
| **1. Roofline** | §1.1 | Reads `sm__throughput` + `gpu__compute_memory_throughput` → classifies COMPUTE_BOUND / MEMORY_BOUND / BALANCED |
| **2. Characterise** | §1.2 / §1.3 | Memory-bound → dives into DRAM, L2, L1 metrics. Compute-bound → checks Tensor Core vs FMA utilisation |
| **3. Anomaly Scan** | §1.4 / §1.5 | Detects: LOW_OCCUPANCY, BANK_CONFLICT, WARP_DIVERGENCE, EXCESSIVE_GLOBAL_LOADS, TENSOR_CORES_UNUSED |
| **4. Map to Code** | §1.6 | Locates `.cu` source, scans for missing `__shared__`, absent `#pragma unroll`, idle Tensor Cores, etc. |

After the 4-step analysis, the LLM synthesises all findings (metrics + source code + code issues) into a professional report with line-level recommendations.

### Sample Kernels Included

| Kernel | Description | Key Characteristics |
|--------|------------|-------------------|
| `matmul_naive.cu` | Naive N×N matmul | No shared memory, no unrolling, no Tensor Cores |
| `matmul_tiled.cu` | Tiled matmul | `__shared__` memory, TILE_SIZE=32, `#pragma unroll` |
| `matmul_tensor.cu` | WMMA Tensor Core | FP16→FP32, 8-warp (4Y×2X), shmem staging, 19.4 TFLOPS @ N=4096 |

## LLM Integration

The agent uses any **OpenAI-compatible** LLM endpoint for:

1. **Autonomous probe generation** — the LLM writes all 6 CUDA micro-benchmark `.cu` source files from design specifications at runtime; sources are cached in `build/` and reused on subsequent runs
2. **Semantic target resolution** — mapping non-standard metric names to probes (~100ms)
3. **Anomaly analysis** — deep reasoning about detected anomalies (frequency locking, SM masking)
4. **Engineering reasoning** — LLM-authored `_reasoning` and `_methodology` narratives
5. **Kernel bottleneck reports** — professional analysis reports with source-code-level recommendations

All LLM calls have **graceful fallback** to template-based text if the API is unavailable.
